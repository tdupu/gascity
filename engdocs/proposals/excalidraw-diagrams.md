---
title: "Adopt Excalidraw as Standard Diagramming Tool"
---

## Context

The project has no established standard for diagrams. Three tools were evaluated side-by-side across four representative diagram types to pick one. The evaluation covered:

- **A. Simple flowchart** — basic shapes, branching, labels
- **B. System architecture** — containers, grouping, dense layout (browser → CDN → LB → 2 app servers → DB + replica + Redis)
- **C. OAuth 2.0 sequence** — interaction diagrams with multiple actors
- **D. ER data model** — structured entity shapes with fields and relationships

The three candidates: **Mermaid (ELK)**, **D2**, and **Excalidraw**.

## Findings

### Mermaid

- **Source verbosity (B):** 22 lines inline in MDX, no committed file.
- **Build step:** None — rendered live by Mintlify.
- **Strengths:** Zero toolchain friction, native sequence and ER support, inline in docs.
- **Weaknesses:** Output is stiff and impersonal; per-node styling requires verbose `style` blocks; the ELK renderer directive applies only to flowcharts (sequence and ER diagrams get the default renderer); colour control over subgraphs is coarse.

### D2

- **Source verbosity (B):** 30 lines; 36 in sketch mode.
- **Build step:** `d2 --layout=elk` → SVG; committed output.
- **Rendered output (B):** 31 KB SVG (standard), 126 KB (sketch mode).
- **Strengths:** Native sequence support; sketch mode produces a recognisably hand-drawn look; ELK layout is solid.
- **Weaknesses:** Sketch mode SVGs are very large (4× standard); the hand-drawn aesthetic is inconsistent — sketch applies to shapes but not to text rendering; colour customisation uses a different syntax from Mermaid and is less discoverable; no native ER table shape.

### Excalidraw

- **Source verbosity (B):** 478 lines of JSON (`.excalidraw` format, not author-written — produced by the MCP authoring tool).
- **Build step:** `@swiftlysingh/excalidraw-cli` → SVG via `make diagrams-excalidraw`; committed output.
- **Rendered output (B):** 28 KB SVG — smallest of the three.
- **Strengths:** Genuinely hand-drawn aesthetic that is consistent across all shape types; pastel palette is visually expressive and ports cleanly to Mermaid and D2 for parity; authored conversationally via the Excalidraw MCP server without writing JSON by hand; SVG output is compact despite the large source file.
- **Weaknesses:** No native sequence syntax (swim-lane approximation demonstrated for C — acceptable); no native ER table shape (hand-built approximation demonstrated for D — acceptable); `.excalidraw` JSON source files are large and not human-readable.

### Head-to-head for diagram B

| Tool | Source lines | Build step | SVG size |
| --- | --- | --- | --- |
| Mermaid (ELK) | 22 (inline) | No | — (inline) |
| D2 | 30 | Yes | 31 KB |
| D2 (sketch) | 36 | Yes | 126 KB |
| Excalidraw | 478 (generated) | Yes | 28 KB |

The source line count for Excalidraw is misleading — the JSON is generated by the MCP tool, not authored by hand. The authoring effort is comparable to D2 and lower than Mermaid for complex diagrams.

## Decision

**Adopt Excalidraw as the standard for all new diagrams.**

The visual output is the best of the three, and the pastel palette established during this evaluation provides a consistent, readable colour language. The toolchain is already set up (MCP server, CLI renderer, Makefile targets). The two missing native diagram types (sequence, ER) have acceptable approximations, and those diagram types are rare enough that Mermaid inline remains a reasonable fallback when exact syntax is needed.

## Limitations and mitigations

| Limitation | Mitigation |
| --- | --- |
| No native sequence syntax | Use swim-lane approximation (demonstrated in diagram C). For complex sequences where precision matters more than aesthetics, fall back to Mermaid inline. |
| No native ER table shape | Use colour-banded header + text-row approximation (demonstrated in diagram D). |
| `.excalidraw` JSON source is large and opaque | Source files are committed for editability (in excalidraw.com or via MCP) — they are not intended to be read directly. Rendered SVGs are the review artefact. |

## Toolchain

- **MCP server** — `https://mcp.excalidraw.com` configured in `.mcp.json`. Lets Claude author and edit `.excalidraw` files conversationally.
- **CLI renderer** — `@swiftlysingh/excalidraw-cli` run via `npx` (no global install, Node 18+ required). Converts `.excalidraw` → SVG.
- **Makefile target** — `make diagrams-excalidraw` renders changed files under `docs/diagrams/excalidraw/` to `docs/diagrams/excalidraw-rendered/`.

Full setup instructions (including a copy-paste prompt for bootstrapping in a fresh session) are in [`engdocs/contributors/excalidraw-setup.md`](../contributors/excalidraw-setup.md).

## Style guide

The shared pastel palette is defined in [`engdocs/contributors/excalidraw-setup.md § Style guide`](../contributors/excalidraw-setup.md#5-style-guide--shared-palette). All new diagrams must use it so recurring concepts (entry points, compute, persistence, errors) get consistent colour treatment across the docs.

## Authoring workflow

1. Use the `mcp__excalidraw__*` tools in Claude to compose or edit the diagram.
2. Export with `mcp__excalidraw__export_to_excalidraw` → writes `docs/diagrams/<name>.excalidraw`.
3. Run `make diagrams-excalidraw` → generates `docs/diagrams/excalidraw-rendered/<name>.svg`.
4. Commit both the `.excalidraw` source and the rendered SVG.
5. Reference the SVG from any Markdown/MDX page:
   ```markdown
   ![Alt text](/diagrams/excalidraw-rendered/<name>.svg)
   ```

## Migration

Existing **Mermaid inline** diagrams in docs are left as-is. There is no mandatory migration; Mermaid is still acceptable for simple or sequence diagrams where the inline-no-build-step advantage outweighs the aesthetic difference. Migrate to Excalidraw when a diagram is being significantly revised anyway.

No D2 diagrams exist outside the comparison directory — nothing to migrate.
